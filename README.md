سأقسم وظائف الـ Middleware إلى وحدات منطقية مترابطة، بحيث:

كل وحدة تمثل نطاق مسؤولية واضح

كل بند فيها قابل للاختبار

ويمكن تحويلها لاحقاً إلى Test Cases

المعمارية المفترضة:

WhatsApp ↔ Chatwoot ↔ Middleware ↔ Store API

والـ Middleware هو:
Business Orchestration + Session Engine فقط.

🧩 القسم 1 — تكامل Chatwoot (Integration Layer)
الهدف

ضمان أن Middleware يتواصل مع Chatwoot فقط عبر API/Webhook دون تكرار وظائفه.

شروط القبول

✔ يستقبل Webhook من Chatwoot عند وصول رسالة جديدة
✔ يتحقق من صحة التوقيع (إن وجد)
✔ يربط الرسالة بـ account_id الصحيح
✔ يستطيع إرسال رسالة عبر Chatwoot API
✔ يستطيع إرسال Template عبر Chatwoot API
✔ يستطيع إرسال Interactive Message عبر Chatwoot API
✔ لا يتصل مباشرة بـ WhatsApp Cloud API

فشل أي بند = فشل طبقة التكامل.

🧠 القسم 2 — إدارة الجلسات (Session State Engine)
الهدف

إدارة حالة العميل داخل سيناريو تفاعلي.

شروط القبول

✔ عند أول تفاعل يتم إنشاء session جديدة
✔ يتم تخزين:

account_id

contact_id

current_state

last_message_type

metadata (json)

✔ يتم تحديث الحالة بعد كل خطوة
✔ إذا انقطعت المحادثة يمكن استئنافها
✔ انتهاء الجلسة بعد مدة inactivity محددة
✔ دعم أكثر من سيناريو مختلف (Flow A / Flow B)

اختبار عملي:

عميل يدخل قائمة منتجات

يختار منتج

يخرج

يعود

يستكمل من آخر نقطة

🛍 القسم 3 — تكامل المتجر (Store Integration)
الهدف

جلب بيانات ديناميكية من المتجر.

شروط القبول

✔ يستطيع Middleware الاتصال بـ Store API
✔ يجلب قائمة منتجات
✔ يبني رسالة Interactive List ديناميكية
✔ عند اختيار منتج:

يجلب بياناته

يرسل تفاصيله عبر Chatwoot
✔ يدعم معالجة أخطاء Store API
✔ لا يوقف النظام إذا تعطل المتجر

اختبار:

قطع اتصال المتجر

يجب إرسال رسالة fallback للعميل

📋 القسم 4 — الرسائل التفاعلية المتقدمة
الهدف

إدارة منطق الرسائل التفاعلية غير المدعومة من Chatwoot كمنطق.

شروط القبول

✔ يستطيع إنشاء:

List Messages

Button Messages

Multi-step Flows

✔ عند اختيار عنصر:

يتم تفسير الرد

يتم تحديث session

يتم إرسال الرد التالي الصحيح

✔ يدعم ترتيب رسائل متعددة ضمن Flow واحد
✔ يدعم سيناريو “رجوع للخلف”

اختبار:

Flow منتجات

Flow اشتراك تسويق

Flow طلب دعم

📢 القسم 5 — إدارة Opt-in والتسويق
الهدف

إدارة موافقات التسويق.

شروط القبول

✔ تخزين حالة opt_in لكل جهة اتصال
✔ منع إرسال قالب تسويقي بدون opt_in
✔ تسجيل تاريخ الموافقة
✔ دعم سحب الموافقة
✔ دعم تحقق قبل أي Campaign

اختبار:

Contact بدون opt_in

محاولة إرسال Template

يجب رفض العملية

🧮 القسم 6 — قواعد الأعمال (Business Rules Engine)
الهدف

تحديد ماذا يحدث في كل حالة.

شروط القبول

✔ تحديد Flow حسب Trigger (كلمة مفتاحية مثلاً)
✔ دعم قواعد شرطية (إذا اختار X → افعل Y)
✔ دعم قواعد زمنية
✔ منع تداخل Flows
✔ إعطاء أولوية للتدخل البشري (Agent Override)

اختبار:

Agent يتدخل

يجب إيقاف الـ Flow مؤقتاً

🗂 القسم 7 — تخزين البيانات (Persistence Layer)
الجداول المطلوبة (MVP)

✔ accounts
✔ contacts
✔ sessions
✔ session_events
✔ opt_ins
✔ flow_definitions (اختياري MVP+)

شروط القبول

✔ لا يوجد تضارب بيانات
✔ كل Session مرتبطة بـ account
✔ لا يوجد تسرب بين العملاء (Multi-tenant isolation)

🔐 القسم 8 — العزل متعدد العملاء (Multi-Tenant Isolation)
الهدف

ضمان عدم اختلاط بيانات العملاء.

شروط القبول

✔ كل طلب يحمل account_id
✔ لا يمكن لعميل رؤية بيانات عميل آخر
✔ عزل كامل في الجداول
✔ اختبارات Cross-account access

⚠ القسم 9 — إدارة الأخطاء والاستقرار
الهدف

منع انهيار النظام.

شروط القبول

✔ Logging لكل حدث
✔ Logging للأخطاء
✔ Retry عند فشل إرسال رسالة
✔ Timeout للاتصالات الخارجية
✔ لا يؤدي خطأ واحد إلى كسر باقي النظام

🧪 القسم 10 — اختبارات التدفق الكامل (End-to-End Validation)

يجب أن ينجح النظام في:

1️⃣ استقبال رسالة
2️⃣ إنشاء Session
3️⃣ إرسال قائمة منتجات
4️⃣ استلام اختيار
5️⃣ جلب بيانات المنتج
6️⃣ إرسال التفاصيل
7️⃣ تسجيل الأحداث
8️⃣ الحفاظ على الحالة

📌 التعريف النهائي لاجتياز النظام

النظام يعتبر “جاهز MVP” إذا:

جميع الأقسام 1 → 6 تعمل بدون أخطاء

Multi-tenant isolation مثبت

يمكن تشغيل 3 Flows مختلفة بنجاح

لا يوجد اتصال مباشر بـ WhatsApp

يمكن إعادة تشغيل السيرفر دون فقد Session فعالة
